# WhatsApp Payment Links Template — Template Structure Spec

**Date:** 2026-06-27
**Status:** Draft
**Audience:** Internal product team
**Scope:** Payment Links template style within the WhatsApp node

---

## What This Is

A payment links template lets a seller send a WhatsApp message with a button that takes the customer directly to a payment page to complete a specific payment. The payment link is generated by the seller's connected payment provider (Razorpay, PayU, Cashfree, etc.) and delivered through WhatsApp as a native payment action — not just a URL.

Use cases: collect payment for an invoice, request payment for a custom order, send a personalised payment request for a specific amount, follow up on an overdue payment.

**How this fits alongside other payment templates:**

| Template | When to use |
|---|---|
| Payment Links | Seller generates the payment link externally and sends it via WhatsApp. No order summary shown. |
| Order Details | Seller has a fully formed order with line items and wants to show the summary before collecting payment. |
| Checkout Button | Customer has a pending cart and the seller wants them to complete checkout and create an order. |

**Important constraints:**
- **India only.** Tied to WhatsApp Pay and Indian payment providers.
- **Requires Meta approval.** Template shell must be submitted and approved before use.
- **Utility category only.** Must relate to a real, specific payment request.

---

## What the Seller Fills In

### Part 1 — Template shell (created once, submitted for approval)

| Field | Required | What the seller fills in |
|---|---|---|
| Template name | Yes | Unique internal name. Lowercase letters and underscores only. |
| Language | Yes | Language the message is written in. |
| Header | No | Optional text or image above the body. |
| Body | Yes | The main message text. Can include personalisation placeholders (e.g. customer name, amount, description). Max 1,024 characters. |
| Footer | No | Small grey text below the message. Max 60 characters. No personalisation. |
| Button label | Yes | The text on the payment button (e.g. "Pay now", "Complete payment", "Pay ₹{{amount}}" ). Max 20 characters. |

---

### Part 2 — Payment data (populated at send time, per customer)

The seller maps these fields to customer and order variables from the journey. They resolve to real values when the message is sent.

| Field | Required | Notes |
|---|---|---|
| Payment link URL | Yes | The payment page URL generated by the connected payment provider. This is the destination the customer is taken to when they tap the button. |
| Amount | Yes | The payment amount in rupees. Shown to the customer before they tap. |
| Payment reference ID | Yes | A unique ID for this payment request. Returned in the payment confirmation so the seller can match the payment to the right record. |
| Description | No | A short label for what the payment is for (e.g. "Invoice #1042", "Custom order deposit"). Shown on the payment screen. Max 60 characters. |
| Link expiry | No | How long the payment link remains active. After expiry, the button in the message stops working. |

---

## What the Customer Sees

The customer receives the message with:
- The body text (e.g. "Your payment of ₹2,500 is due for order #1042")
- A payment button at the bottom

When the customer taps the button, they are taken to the payment provider's hosted payment page where they can pay via UPI, saved card, or net banking. The experience may be in-app (via WhatsApp Pay) or open in a browser depending on the provider and Meta's integration level.

---

## What Comes Back After the Customer Acts

| Event | What the platform receives |
|---|---|
| Customer completes payment | Payment confirmation with transaction ID, amount paid, payment method, payment reference ID, and timestamp |
| Customer opens link but does not pay | No event from Meta — the payment provider may send a separate webhook depending on integration |
| Customer ignores the message | No event |
| Payment link expires | No event from Meta — button becomes inactive |

The payment reference ID set at send time is returned in the confirmation, allowing the platform to match the payment to the right customer and journey.

---

## Key Constraints

- **India only.** Works with Indian payment providers and WhatsApp Pay infrastructure.
- **Meta approval required.** The template shell is approved once. The payment link and amount are dynamic.
- **Utility only.** Cannot be used for promotional payment collection (e.g. "buy this product"). Must be for a specific, real payment request.
- **Payment link must be pre-generated.** The seller's journey must call the payment provider's API to create the payment link before this node sends the message. The platform does not generate the link — it passes one through. This means the payment provider integration (e.g. Razorpay node) typically runs before this node in the journey.
- **Reference ID must be unique per send.** Reusing the same reference ID across customers will cause matching errors.
- **Link expiry is provider-controlled.** The payment provider sets when the link expires. The seller's expiry field is advisory — actual expiry depends on the provider's configuration.

---

## Relationship to the Razorpay Node

The Razorpay node and this template work together in sequence:

1. **Razorpay node** — generates the payment link and stores it as `response.paymentLink`
2. **WhatsApp Payment Links template** — sends the message with `response.paymentLink` mapped to the Payment link URL field

The WhatsApp Payment Links template is the delivery mechanism. The Razorpay node (or equivalent payment provider node) is the generation step. Neither replaces the other.
